## 1. Context — למה התוכנית הזו ומה היא משיגה

הפרויקט מחליף קובץ Excel מורכב לחישוב וניהול שכר (IAI/MALAT). ה‑PRD וה‑`CLAUDE.md` הגדירו במקור בסיס קוד **Flutter יחיד**. בעקבות החלטת המשתמש, התוכנית **מיישמת קודם אפליקציית WEB פשוטה ב‑HTML/JS**, ורק לאחר מכן ניגשים לנייד.

**החלטות מפתח (מחייבות את התוכנית):**

1. **גרסת ה‑WEB היא קבועה ומקבילה — לא אב‑טיפוס חד‑פעמי.** היעד הסופי (אושרר ע"י המשתמש 2026‑07‑01): אפליקציית WEB פשוטה **ללא התקנה** למחשב בעבודה, שתסתנכרן עם אפליקציית מובייל שתעשה את אותו הדבר. מתחילים ב‑WEB כי קל יותר לנטר בו את הפעילות. שתי הגרסאות חולקות **אותו פורמט JSON** כמקור אמת.
2. **כמה שיותר פשוט:** **vanilla HTML/CSS/JS, ללא framework, ללא שלב build**. התלות החיצונית היחידה היא `SheetJS` (Excel), vendored מקומית (לא CDN) — הכול עובד offline ונפתח בקליק.
3. **קובץ ה‑Excel המקורי סופק** ופוענח (WP0.2); הנוסחאות ו‑golden cases מתועדים.
4. **שלב הנייד מחוץ להיקף** התוכנית הזו (פרק בסוף, רמת‑על בלבד).

**התוצאה הרצויה:** אפליקציית WEB סטטית, נפתחת בקליק, עברית RTL, ₪, מצב כהה/בהיר, מנוע חישוב מדויק (≤ ₪1 מול האקסל), כל המודולים (נוכחות, משוער, בפועל, הפחתות, קרן עזרה, היסטוריה+גרפים, הגדרות), ייצוא/ייבוא JSON ו‑Excel, וטעינה/שמירה מול OneDrive דרך File System Access API.

## WP10.2 — קליטת תאריכים סטנדרטיים בייבוא Excel — S
**הערת משתמש:** "לשנות קליטת היסטוריה (מאקסל) לתאריך סטנדרטי."
**ממצא (באג):** (א) בעמודות תאריך ("נוכחות", "קרן עזרה") — תאי תאריך אמיתיים (serial) כן מומרים (`excelSerialToISODate`, `excel-io.js:170-174`), אבל טקסט חייב להיות בדיוק `YYYY-MM-DD` (regex ב‑415, 524) — `15/06/2026` או `15.6.2026` מפילים את השורה ("תאריך לא תקין"). (ב) הגרוע ביותר — עמודת "חודש" ב"סטטוס חודשי" נקראת כ‑String גולמי **ללא** נרמול serial (`excel-io.js:467-469`): כשמקלידים `2026-06` ואקסל ממיר אוטומטית לתא תאריך, הייבוא רואה מספר serial ונכשל ("חודש לא תקין") — למרות שהמשתמש עקב אחר הוראות התבנית (309). (ג) הייצוא/תבנית כותבים תאריכים כטקסט ISO, לא תאי תאריך — עריכה חוזרת באקסל מפעילה את מלכודת ההמרה.
**צעדים (כולם ב‑`src/io/excel-io.js`, שכבת I/O בלבד):** (1) הרחבת `normalizeDateCell`: גם אובייקטי `Date` וגם טקסט `dd/mm/yyyy` ו‑`dd.mm.yyyy` (יום‑קודם, תקן ישראלי) → `YYYY-MM-DD`. (2) `normalizeMonthCell` חדש לעמודת חודש: serial → `YYYY-MM` (דרך `excelSerialToISODate().slice(0,7)`), טקסט `YYYY-MM` או `MM/YYYY` → `YYYY-MM`. (3) עדכון גיליון ההוראות בתבנית. (4) בדיקות יחידה לכל פורמט. (5) אופציונלי: ייצוא/תבנית עם תאי תאריך אמיתיים (`cellDates`) בפורמט תצוגה `dd/mm/yyyy` — הייבוא כבר יודע serial, כך ש‑round‑trip נשאר בטוח.
**קבלה:** קובץ שבו אקסל המיר תאריכים/חודשים לתאי תאריך נטען ללא שגיאה; `15/06/2026` טקסט נקלט; round‑trip ייצוא→ייבוא ללא רגרסיה (`node --test`).

## WP10.8 — כוננות פר יום: ערך‑יום × ימי נוכחות — M ✅ אומת אדוורסרית (סבב ב')
**הערת משתמש:** "לוודא כוננות פר יום." **המודל שהוכרע (סבב ב'):** כוננות ניתנת על **כל יום נוכחות בפועל** (יום עם `start != null`, כולל יום חלקי שהושלם בחופשה/מחלה; **לא** כולל יום חופשה/מחלה/השתלמות טהור ללא נוכחות). שכר הכוננות החודשי = **ערך יום כוננות × מספר ימי הנוכחות בחודש**. *(זה מחליף את הצעת סבב א' של P16×P17 לפי שעה.)*
**ממצא:** כוננות היום = רק רכיב שכר קבוע `duty` (`defaults.js:143`: `inTax✓ inNI✓ inPension✓ inTraining✗`, `appliesToGrades:['all']`) — סכום ידני, ללא כל גזירה מימי נוכחות. המנוע כבר מקבל את `month.days` (`engine.js:83`) ובונה `enrichedDays` (90‑101), ויש תקדים לספירת ימים לפי predicate (`overtimeDays = enrichedDays.filter(...).length`, `engine.js:110`). סיווג "נוכח" קיים ב‑UI: `start != null && leave.type !== 'training'` (`attendance.js:290-291`). `docs/excel-formulas.md:394` מאשר שכוננות בפנסיה אך לא בקה"ש — לכן `duty`‑flags הם הנכונים.
**צעדים:** (1) **הגדרות:** שדה `personal.standbyDayValue` (₪ ליום כוננות; ברירת מחדל 0 = כבוי, אין השפעה — כלל #4 אין hard‑coding) ב‑`schema.js` (ולידציה ≥0, כמו `overtimeHourValue`) + שורת `numG` ב‑`settings.js` + מחרוזת "ערך יום כוננות (₪)". (3) **UI:** `standbyPay` כשורה בפירוט הברוטו (WP10.4) + מחרוזת "כוננות". (4) golden case חדש (ערך‑יום × ימי נוכחות). snapshots שמורים לא נגעים (כלל #6).
**קבצים:** `src/model/schema.js`, `src/engine/engine.js`, `src/ui/settings.js`, `src/ui/strings.he.js`, `src/ui/estimate.js`, `test/`.
**החלטה — רכיב `duty` הקבוע:** **דו‑קיום, לא החלפה** — `duty` נשאר לתאימות לאחור (golden cases + מי שהזין סכום קבוע). **אזהרת כפל:** אם `standbyDayValue>0` וגם `duty.amount>0`, שניהם ייספרו — להוסיף hint ב‑`settings.js`: "השאר את כוננות הקבועה = 0 אם אתה משתמש בערך יום כוננות".

## 8.1 מפרט מנוע הנוכחות (מקור אמת — לא לשנות ללא אימות מול האקסל)

**קטגוריזציית שעות יומית** (`categorizeDay`), קלט: `start S`, `end E`, `breakCode`, `dow`:
```
unapproved   = max(0, min(E, approvedStartTime) − S)      // לפני 06:30
aStart       = max(S, approvedStartTime)
quotaLine    = aStart + fullDayHours                        // 07:00+8:54 = 15:54
breakOverlap = חיתוך(breakWindow(code), [aStart, min(E, quotaLine)])
regularPaid  = (min(E, quotaLine) − aStart) − breakOverlap  // עד 8:24
excess       = max(0, E − quotaLine)

excess == 0            → רגיל בלבד
0 < excess ≤ 1h        → zero += min(E,17:00)−quotaLine ; unapproved += E−max(quotaLine,17:00)
excess > 1h            → overtime = excess (כולל השעה הראשונה)
שישי (fridayAllOvertime) → overtime = (E−aStart)−breakOverlap ; regular=zero=0
```
פרמטרים (ב‑`settings.national.attendanceParams`, ניתנים לעריכה): `approvedStartTime=6.5`, `fullDayHours=8.9`, `halfDayHours=4.45`, `zeroHourCutoff=17.0`, `firstBandHours=1.0`, `fridayAllOvertime=true`, `defaultBreakCode`, `breakWindows[]`.

**דוגמאות מאומתות:** 07:00→17:30 (הפסקה 12:00–12:30) ⇒ רגיל 8:24, ש"נ 1:36 ✓ · 07:30→17:20 ⇒ אפס 0:36 + ללא‑אישור 0:20 ✓.

**השלמת חיסורים חודשית** (`calcMonthlyShortfall`): חיסור = `fullDayHours − presenceInQuota` לימים עם נוכחות ≥ חצי יום; כיסוי בסדר **אפס → ש"נ → ללא‑אישור → ירידת שכר**; ירידת שכר → התראה אדומה; מוצג % ניצול אפס. ימי חופשה/מחלה/השתלמות נספרים כעבודה (לא יוצרים חיסור). שישי לא יוצר חיסור.
