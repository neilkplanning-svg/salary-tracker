## 2. נוסחאות שעות

### 2.1 שעות רגילות (עמודה E)

```excel
=IF(C3="שישי", 0,
  IF(D3>=6.5/24,
    IF(E3>12/24,
      IF(E3-D3>8.9/24,
        8.4/24,
        IF(E3-12/24>0.5/24, E3-D3-0.5/24, 12/24-D3)),
      E3-D3),
    0))
```

**קבועים:**
- `8.4h` = יום עבודה מלא
- `8.9h` = סף שעות נוספות
- `6.5h` = שעת כניסה מינימלית (06:30)
- `0.5h` = הפסקת צהריים (מנוכה אם יציאה אחרי 12:00)

**פרשנות:** שישי = 0 שעות רגילות. ימים אחרים: עד מקסימום 8.4 שעות, בניכוי 30 דקות הפסקה אם יציאה אחרי 12:00.

### 2.2 שעות אפס (עמודה F)

```excel
=IF(AND(E3=8.4/24, D3-C3<=10/24, D3-C3>8.9/24),
  IF(D3<16.9/24,
    D3-C3-8.9/24,
    IF(16.9/24-C3>8.9/24, 16.9/24-C3-8.9/24, 0)),
  0)
```

שעות "אפס" = יום מלא (8.4 שעות רגילות) + עבודה בין 8.9 ל-10 שעות → החלק מעל 8.9 שעות.

### 2.3 שעות נוספות (עמודה G)

```excel
=IF(C3="שישי",
  D3-C3,
  IF(IF(F3>0, 0, IF(D3-C3>8.9/24, D3-C3-8.9/24, 0)) < 1.1/24,
    0,
    IF(F3>0, 0, IF(D3-C3>8.9/24, D3-C3-8.9/24, 0))))
```

**פרשנות:**
- שישי: כל השעות = ש"נ
- ימים אחרים: ש"נ = total - 8.9h, אם total > 8.9h
- אם קיימות "שעות אפס" → ש"נ = 0
- אם ש"נ < 1.1h (סף מינימלי) → ש"נ = 0

## 3. שעות נוספות — חישוב תשלום

### 3.1 שעות מאושרות

```
P12 = M14 × 24           ← שעות ש"נ מאושרות
P19 = COUNTIF(H3:H33,">0") ← מספר ימים עם ש"נ
```

### 3.2 גורם הכפלה (P18)

```
P18 =
  IF(OR(P12<=11.9, AND(P12>11.9, P19<6)),  1.00,
  IF(AND(P12>=11.9, P12<=19.9, P19>=6),    1.25,
  IF(AND(P12>19.9, P19>=6),                1.30, 0)))
```

| ש"נ מאושרות | ימי ש"נ | גורם |
|------------|---------|------|
| ≤ 11.9 שעות | כל מספר | 1.00 |
| > 11.9 שעות | < 6 ימים | 1.00 |
| 11.9–19.9 שעות | ≥ 6 ימים | 1.25 |
| > 19.9 שעות | ≥ 6 ימים | 1.30 |

### 3.3 נוסחת תשלום — **לוגריתם בסיס 8** ⚠

```
P20 = IFERROR(
  (P12 × LOG(P12, 8)) × P18
  + P13
  + (P14 × P15)
  + (P16 × P17),
  0)
```

| משתנה | משמעות |
|-------|--------|
| P12 | שעות מאושרות |
| LOG(P12,8) | לוגריתם בסיס 8 (=ln(P12)/ln(8)) |
| P18 | גורם הכפלה |
| P13 | שעות 100% (בונוס ספציפי) |
| P14 | פרמיה שבת/חג |
| P15 | שעות שבת/חג |
| P16 | פרמיה כוננות |
| P17 | שעות כוננות |

**P21 = 76.23 ₪/שעה** (ערך שעה נוספת)

**חישוב תשלום:**
```
N5 = P20 × 76.23        ← ש"נ ברוטו
M5 = N5 × 0.988         ← ש"נ נטו (לאחר קיזוז 0.988)
```

**דוגמה — יוני 2026 (24 שעות, 7 ימים, גורם 1.3):**
```
LOG(24, 8) = ln(24)/ln(8) = 3.1781/2.0794 = 1.5286
P20 = 24 × 1.5286 × 1.3 = 47.684
N5  = 47.684 × 76.23 = 3,634.92 ₪
M5  = 3,634.92 × 0.988 = 3,591.30 ₪
```

**⚠ אל תבלבל עם 125%/150% פשוטים — הנוסחה היא לוגריתמית!**

## 4. חישוב ברוטו (M17)

```
attendance_days = J34 = COUNTIF(J3:J33, "V")

N2 = (קבוע אישי) + (תוספת ליום נוכחות) × attendance_days
M2 = N2 × 0.988          ← שכר יסוד + כוננות

M3 = (ערך אישי) × 0.988  ← שכר עידוד

M4 = 3879 + (ערך אישי)   ← רכב + טלפון (ללא 0.988!)

M5 = N5 × 0.988              ← שעות נוספות (ראו §3)

M6 = (ערכים אישיים)          ← תוספות (ללא 0.988!)

M17 = M2 + M3 + M4 + M5 + M6
```

**⚠ גורם 0.988:** מוחל על M2, M3, M5 — לא על M4 (רכב/טלפון) ולא על M6 (תוספות).

## 6. מס הכנסה (V13)

### בסיס חייב
```
M18 = M17 + P9   ← gross + imputations
```

### ⚠ מינימום מדרגה ראשונה — פרט קריטי

```
S6 = V33 = V31 + V32
```

`V33 = pension_employee + training_employee` משמש כ**תחתית המדרגה הראשונה** — הכנסה עד לסכום זה פטורה ממס. ערך זה דינמי ומשתנה עם הברוטו.

### מדרגות מס (2026)

| מינימום | מקסימום | שיעור |
|---------|---------|-------|
| V33 (דינמי ~2,182) | 7,010 | 10% |
| 7,011 | 10,060 | 14% |
| 10,061 | 19,000 | 20% |
| 19,001 | 25,100 | 31% |
| 25,101 | 46,690 | 35% |
| 46,691 | 60,130 | 47% |
| 60,131 | 150,000 | 50% |

**נוסחה לכל מדרגה (תאי V6:V12):**
```excel
V_i = IF(M18 > T_i, (T_i - S_i) × U_i, IF(M18 > S_i, (M18 - S_i) × U_i, 0))
```
שקול ל: `V_i = clamp(M18 - S_i, 0, T_i - S_i) × rate_i`, כאשר `S_i = T_{i-1} + 1`.

**⚠ פער ₪1 בין מדרגות — לא לתקן ל"רציף"!** מינימום כל מדרגה = מקסימום הקודמת **+ 1**
(S7=7011, S8=10061, S9=19001, S10=25101...). זהו מבנה האקסל המקורי. שימוש במדרגות רציפות
(7010, 10060...) מנפח את המס בכ-₪1 ומפיל את `net` מעל סף הסטייה — היה שורש באג WP1.2.
המנוע משחזר את המבנה הזה (`defaults.js`, `golden-cases.json`).

### נקודות זיכוי
```
credit = (כמות נק' זיכוי אישית) × 242
V13 = max(0, SUM(V6:V12) - credit)
```

### נוסחאות הבסיסים (אקסל מחקר, גיליון "מחקר 2015")
```
ברוטו שוטף (E13)   = Σ(יסוד..עידוד) + רכב + טלפון + ביטוחי-רכב + שי-פסח
ברוטו ב.ל  (E14)   = ברוטו שוטף + זקיפות (סלולרי/שיניים/ארוחות)

```

### שיוך לבסיסים (flags)
| רכיב (id) | מס | ב"ל | פנסיה | קה"ש | הערה |
|-----------|:--:|:--:|:--:|:--:|------|
| שכר יסוד (`base`) | ✓ | ✓ | ✓ | ✓ | כל הדירוגים |
| ותק (`seniority`) | ✓ | ✓ | ✓ | ✓ | |
| ותק למחקר (`researchSeniority`) | ✓ | ✓ | ✓ | ✓ | מחקר |
| שונות / שונות א׳ (`misc`/`miscA`) | ✓ | ✓ | ✓ | ✓ | מחקר |
| שכר עידוד (`incentive`) | ✓ | ✓ | ✓ | ✓ | מהנדסים/מחקר |
| השלמת שכר חוזה (`contractSupplement`) | ✓ | ✓ | ✓ | ✓ | חוזה |
| **כוננות (`duty`)** | ✓ | ✓ | **✓** | **✗** | ⚠ בפנסיה אך **לא** בקה"ש — נאמת כ: בסיס פנסיה − בסיס קה"ש = כוננות |
| תוספות לשכר (`salaryAdditions`) | ✓ | ✓ | ✗ | ✗ | מהנדסים |
| טלפון (`phone`) | ✓ | ✓ | ✗ | ✗ | |
| החזר ביטוחי רכב (`carInsurance`) | ✓ | ✓ | ✗ | ✗ | חוזה |
| מחקר דולרי (`researchDollar`) | ✗ | ✗ | ✗ | ✗ | נטו — מתווסף לנטו בלבד |
| **רכב** (toggle נפרד) | ✓ | ✓ | ✗ | ✗ | תוספת מזומן או שווי גילום — `carTaxable` |
| **ש"נ** (תת-מנוע) | ✓ | ✓ | ✗ | ✗ | `overtimePay` |

**זקיפות** (`IMPUTATION_COMPONENTS`, מעמודת P): שי-לפסח, שווי-ארוחות, שווי-שיניים, השתתפות-ארוחות, שווי-סלולרי — כולן `inTax`+`inNI` (ב-M18 ו-94010), לא בפנסיה/קה"ש.

**תיקון "נטו-בלבד" (WP10.10, `engine.js`):** רכיב `earnings[]` שכל 4 ה-flags שלו כבויים (כרגע: `researchDollar` בלבד) **מוחרג** מ-`grossFromEarnings`/`gross` (כי אינו תשלום ברוטו אמיתי — רק ערך שכבר משולם בנטו), ומתווסף ישירות ל-`net` **אחרי** כל הניכויים (`net = gross − מיסים/ניכויים + netOnlyAmount`). כך `gross` המוצג תואם את התלוש (לא מנופח ברכיב שכבר בנטו), ו-`net` נכון כפי שהיה. אין שינוי לרכיבים עם flag פעיל כלשהו.

### כוננות לפי יום נוכחות — `standbyDayValue` (WP10.8, `engine.js`)
בנוסף לרכיב `duty` הקבוע (החודשי, מהקטלוג — נשאר בשימוש למי שמעדיף סכום חודשי קבוע), נוסף ערוץ אדיטיבי נפרד: `personal.standbyDayValue` (₪ ליום כוננות; ברירת מחדל 0 = כבוי, ללא השפעה). המנוע מחשב `presenceDays` — מספר ימי הנוכחות בפועל באותו חודש, לפי אותו היגיון הצגה כמו `attendance.js` (`workedWithCompletion`):

```
presenceDays = ימים עם d.start != null && d.leave?.type !== 'training' && !d.training
standbyPay   = personal.standbyDayValue × presenceDays
```

יום נספר כ"נוכחות" גם אם הושלם ליום מלא מחופשה/מחלה (`d.start` קיים), אך **לא** נספר יום היעדרות טהורה (חופשה/מחלה/השתלמות ללא נוכחות בפועל, `d.start == null`) וגם לא יום השתלמות. `standbyPay` מקבל בדיוק את אותם base-flags כמו `duty`: נכנס ל-`gross`, ל-`niBase`/`taxableIncome` (מס+ב"ל), ול-`pensionBase` — אך **לא** ל-`trainingFundBase`. מיושם רק במסלול ה-NEW (earnings[]); אינו נוגע במסלול ה-LEGACY. אם המשתמש מזין גם `standbyDayValue` וגם משאיר את רכיב `duty` הקבוע ≠ 0 — הכוננות תיספר פעמיים (ראו אזהרה במסך ההגדרות).

### תקרת קה"ש + השלמה (נק' 6)
`trainingBase > 15712` ו-`depositAboveCap=false` → עובד מנכה על התקרה (15712×2.5%=392.80), וחלק המעסיק מעבר לתקרה `(trainingBase−15712)×7.5%` מתווסף לברוטו ולבסיס המס/ב"ל כ"השלמת ק.השתלמות" (`trainingComplement`). מאומת מול תלוש מהנדסים (ראו golden case `engineer-05-2026`).

### עיצוב גנרי (החלטת המשתמש)
המשתמש **אינו** מזין/עורך flags. הקטלוג קבוע (אני קובע את השיוך מהמקורות); המשתמש מזין **סכום** לכל רכיב, 0 = הרכיב לא בדירוגו. כך אותו טופס גנרי משרת את כל הדירוגים (מהנדסים/חוזה/מחקר).

### WP8.8b — פנסיה שנייה על נלווים (OI-07) ✅ נפתר
בכל התלושים (כל הדירוגים) יש קרן/ביטוח שנייה על "ברוטו מבוטח נלווה" (שדה `5800`) — נפרד מבסיס הפנסיה העיקרי (`91025`). נוסחה: `5800 × 7% = pension2` (מאומת ≤₪0.01 מול כמה תלושים, כולל קרן "מבטחים נילווים לקיצ"; דוגמאות סינתטיות ב-`golden-cases.json`).
המנוע: `personal.ancillaryPensionBase` (שדה ישיר, לא נגזר מ-`earnings[]` — ניסיון לפרק אותו לרכיבי `earnings` בודדים לא התכנס למספר עגול בין הדירוגים) + `personal.pensionRateEmployee2` (ברירת מחדל 7%) → `pension2 = ancillaryPensionBase × pensionRateEmployee2`, מנוכה מהנטו (`calcPensionSavingsCredit` גם מצרף אותו לבסיס זיכוי החסכון, ראו להלן). `golden-cases.json`: `engineer-05-2026`/`contract-06-2025`/`contract-03-2023` מאמתים `pension2` ≤ ₪0.01.

### WP8.8b — מס הכנסה: החלפת מנגנון V33 בזיכוי חסכון (OI-08) ◐ שופר משמעותית, לא ≤₪1
**הממצא המרכזי:** קריאת רצף תלושים מלא ינואר–מאי 2026 (לא רק חודש בודד) חשפה את מנגנון "הסכומים המצטברים" של MALAT: כל תלוש מציג `זיכוי חסכון`, `הכנסה פטורה ממס`, `סה"כ זיכוי מנקודות` ו-`מס הכנסה שנתי לגביה` כערכים **מצטברים משנת המס**, וה"סכום" בפועל שנגבה החודש = הפרש מהחודש הקודם. שינוי נקודות זיכוי (9.25→11.75, כנראה עקב הולדת ילד) *וגם* שינוי חוק מדרגות המס שפורסם רטרואקטיבית ב-31/3/2026 גרמו שניהם לתיקון רטרואקטיבי בתלוש 05/26 — לכן זה **לא** מדגם נקי לכיול (הוחלט עם המשתמש להשתמש ב-2023/2025 בלבד).

**זיהוי "זיכוי חסכון" (סעיף 45א):** קבוע על פני 3 חודשים בלתי-תלויים (01–05/26, 06/25) ב-**₪237.65/חודש** בדיוק = `35% × min(הפקדת עובד לפנסיה, 679)` — התקרה תמיד חוסמת (הפקדות בפועל ₪1,250–1,800/חודש). מודל: `national.pensionSavingsCreditRate=0.35`, `national.pensionSavingsCreditCap=679`, `calcPensionSavingsCredit(pension+pension2, rate, cap)` ב-`engine.js`.

**"הכנסה פטורה ממס" — לא "הטבה חודשית קבועה" כפי שהונח קודם:** מעקב אחר 5 חודשים רצופים (01–05/26) הראה שהשדה **לא** גדל מדי חודש — הוא קפץ פעם אחת, בדיוק כשמופיע "מענק לידה (נטו)" בתלוש. כלומר מדובר בסכום **חד-פעמי** (מענקים פטורים כמו מענק לידה), לא בפטור שוטף — לכן **לא** מומש כניכוי אוטומטי במנוע; אם יידרש, ייצוג דרך `imputations`/`otherDeductions` עתידי לפריטים ספציפיים.

**החלטה: הסרת מנגנון "מדרגה ראשונה דינמית" (V33 = פנסיה+קה"ש) מהאקסל המקורי.** V33 ו"זיכוי חסכון" הם שני מודלים חופפים לאותה הטבת-מס על הפקדות פנסיה — הפעלת שניהם יחד "מכפילה" את ההטבה ומרחיקה מהמס האמיתי (נבדק: `contract-03-2023` עם V33+זיכוי חסכון היה סוטה משמעותית יותר לחודש; בלי V33, רק עם זיכוי חסכון — הפער קטן משמעותית). לכן `calcIncomeTax` הפסיק לקבל `firstBracketMin` (V33) ומקבל `pensionSavingsCredit` במקום. `golden-cases.json` case `2026-06-full` (מבוסס האקסל המקורי, לא תלוש אמיתי; ערכי golden case סינתטיים) עודכן בהתאם.

**פער שייר לא נפתר (~₪16–29/חודש, `contract-06-2025`/`contract-03-2023`):** גם לאחר ההחלפה, המנוע החד-חודשי (ללא מצב מצטבר — כלל ה-snapshot #6 ב-`CLAUDE.md`) אינו יכול לשחזר את שיטת הניכוי המצטברת-שנתית של MALAT באופן מדויק, במיוחד כש"הכנסה בפריסה שנתית" (פריסת בונוסים/מענקים לצורך מס, סעיף 8 לפקודה) פעילה — מופיעה גם ב-06/2025 (בונוס רבעוני קודם) וגם ב-03/2023. **קריטריון הקבלה של WP8.8ב (≤₪1) לא הושג עבור `incomeTax`/`net` בתלושים בודדים** — שיפור מ-~₪150+/חודש ל-~₪16–29/חודש הוא שיפור משמעותי (פי 5–9), אך דורש state מצטבר לכל השנה כדי לסגור לגמרי, שינוי ארכיטקטוני שאינו בהיקף v1 (מנוגד לכלל ה-snapshot). הרכיבים האחרים (NI, בריאות, פנסיה, פנסיה שנייה, קרן השתלמות) תואמים ≤₪0.1.

**רכיבים פטורי ב"ל (לא נפתר):** לא כל הנלווים/זקיפות חייבי ב"ל (03/24: שווי לימודים + החזרי ביטוח רכב; 06/25: השתתפות בימי גיבוש, סכום קטן פטור ב"ל). המודל הנוכחי: בנייה מ"ברוטו ב.ל" (94010) שבתלוש, לא מפירוק מלא של הרכיבים.

**ניכויים אחרים** (ועד, דמי חבר, קרן עזרה, ביטוח שיניים, הלוואות): קיימים ב-`otherDeductions` — לא נכללו ב-cases (`net` המחושב אינו שווה ל"נטו לבנק" בתלוש בפועל ללא הזנתם; ראו `pension2`/`otherDeductions` ב-golden cases לדוגמת שימוש).

| `approvedStartTime` | 6.5 (06:30) | לפני כן → "ללא אישור" |
| `fullDayHours` | 8.9 (8:54) | נוכחות ליום מלא מ-aStart |
| `halfDayHours` | 4.45 (4:27) | מינימום נוכחות להשלמת חיסור ממאגרים |
| `zeroHourCutoff` | 17.0 (17:00) | גבול אפס/ללא-אישור בשעה הראשונה |
| `firstBandHours` | 1.0 (1:00) | רוחב "שעה ראשונה"; מעל → כל העודף ש"נ |
| `fridayAllOvertime` | true | שישי: כל הנוכחות → ש"נ |
| `breakWindows` | 3 חלונות 30 דק' (11:30–13:00, צומצם 2026-07-01) | `breakCode`=אינדקס; null=ללא |

### אלגוריתם (יום רגיל)

```
unapproved_pre = max(0, min(E, approvedStartTime) − S)
aStart         = max(S, approvedStartTime)
quotaLine      = aStart + fullDayHours
breakDeducted  = overlap(breakWindows[code], [aStart, min(E, quotaLine)])
regularPaid    = (min(E, quotaLine) − aStart) − breakDeducted
excess         = max(0, E − quotaLine)

excess == 0          → רגיל בלבד
0 < excess ≤ 1h      → zero = min(E,17:00)−quotaLine; unapproved_post = max(0,E−17:00)
excess > 1h          → overtimeHours = excess (כולל שעה ראשונה — WORK_PLAN §B החלטה #4)
```

### שישי (`fridayAllOvertime=true`)
```
overtime = (E − aStart) − breakDeducted   // כל הנוכחות המאושרת → ש"נ
regular = zero = 0
// חלון 06:30 והפסקה חלים גם בשישי (WORK_PLAN §B החלטה #2)
```

### דוגמאות (מאומתות מ-WORK_PLAN §B)

| כניסה | יציאה | הפסקה | quotaLine | excess | תוצאה |
|---|---|---|---|---|---|
| 07:00 | 17:30 | 12:00–12:30 | 15:54 | 1:36 > 1h | ש"נ=1:36 |
| 07:30 | 17:20 | ללא | 16:24 | 0:56 ≤ 1h | אפס=0:36, ללא-אישור=0:20 |
